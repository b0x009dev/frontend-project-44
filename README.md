# Brain Games

## Архитектурная характеристика проекта

Настоящий проект представляет собой консольную программную систему учебного назначения, реализующую набор интерактивных интеллектуальных мини-игр с единым сценарием исполнения. Несмотря на сравнительно небольшой объём кодовой базы, внутренняя организация приложения демонстрирует полноценное архитектурное разделение ответственности между уровнями запуска, координации игрового процесса, регистрации доменных модулей и непосредственной генерации игровых данных.

Систему следует рассматривать не как совокупность независимых скриптов, а как компактный модульный CLI-комплекс, в котором выделены:

- слой точек входа командной строки;
- слой оркестрации игрового сценария;
- слой реестра и диспетчеризации игровых модулей;
- слой предметной логики конкретных игр;
- слой сервисных и инфраструктурных функций.

Подобное построение обеспечивает понятность кода, локализацию изменений, расширяемость при добавлении новых игр и снижение дублирования в базовом сценарии взаимодействия с пользователем.

### Архитектурная модель

В основе проекта лежит централизованный сценарий исполнения игры, реализованный в модуле `src/pipeline.js`. Данный модуль выступает в роли управляющего движка и определяет общий жизненный цикл любой игры:

1. приветствие пользователя;
2. разрешение имени игры через реестр;
3. вывод правил;
4. выполнение фиксированного числа игровых раундов;
5. сравнение введённого ответа с эталонным значением;
6. формирование результата завершения игры.

Таким образом, проект организован по модели **единый движок + подключаемые игровые стратегии**. Движок не содержит специализированной логики отдельных игр; его задача состоит в координации процесса, тогда как конкретные игровые модули поставляют правила и данные отдельного раунда.

Такой подход соответствует хорошей инженерной практике: инвариантная часть поведения сосредоточена в одном месте, а вариативные аспекты вынесены в самостоятельные модули. За счёт этого достигается структурная целостность проекта и высокая предсказуемость расширения функциональности.

### Архитектурные паттерны и принципы

#### 1. Template Method в функциональной форме

Общий алгоритм исполнения игры жёстко задаётся модулем `pipeline.js` и не зависит от типа конкретной игры. Для всех игровых режимов неизменными остаются:

- порядок вызовов;
- количество раундов;
- способ вывода вопроса;
- способ чтения ответа;
- механизм проверки результата;
- логика завершения игры.

Изменяемой частью является только генерация правил и набора данных конкретного раунда. Тем самым в проекте реализуется паттерн **Template Method**, но не через иерархию классов, а через композицию функций и модулей.

#### 2. Strategy в модульной реализации

Каждая игра представляет собой отдельную стратегию, предоставляющую общий контракт:

- функцию получения правил игры;
- функцию генерации одного игрового раунда.

Реестр игр в `src/game-registry.js` связывает строковый идентификатор CLI-команды с соответствующей реализацией стратегии. Это позволяет движку работать единообразно с любым игровым режимом, не будучи жёстко связанным с его внутренней логикой.

#### 3. Single Responsibility Principle

Хотя проект не претендует на промышленную многослойную архитектуру, в нём достаточно последовательно соблюдается разделение обязанностей:

- `bin/*` отвечает только за запуск нужной команды;
- `pipeline.js` — за управление жизненным циклом игры;
- `game-registry.js` — за сопоставление имени игры и модуля реализации;
- `games/*` — за предметную логику и генерацию данных раунда;
- `greeting.js` и `game-over.js` — за стартовый и финальный этапы сценария;
- `cli.js` — за получение пользовательского ввода;
- `utils.js` — за общую инфраструктурную функцию генерации случайных чисел.

Подобное распределение ответственности делает код обозримым и методически прозрачным.

#### 4. DRY и централизация повторяющегося сценария

Ключевым архитектурным концептом проекта является устранение дублирования раундовой логики. Каждая игра возвращает объект раунда единого формата, а не реализует самостоятельно взаимодействие с пользователем. За счёт этого общая механика вынесена в `pipeline.js`, что существенно снижает избыточность кода и упрощает поддержку.

### Подробная роль каждого уровня системы

#### Уровень CLI-входа: `bin/`

Файлы в каталоге `bin` представляют собой тонкие точки входа. Каждая команда командной строки выполняет минимально необходимое действие: либо запускает общее приветствие (`brain-games.js`), либо передаёт строковый идентификатор соответствующей игры в `pipeline()`.

Такое решение важно с архитектурной точки зрения по двум причинам:

1. Точки входа не содержат предметной логики.
2. Изменение внутренней реализации игрового сценария не требует модификации внешних CLI-команд.

Иными словами, каталог `bin` формирует интерфейс запуска, а не бизнес-логику приложения.

#### Уровень оркестрации: `src/pipeline.js`

Данный модуль является центральным координатором приложения. Он задаёт инвариантную структуру исполнения любой игры, в том числе:

- количество раундов (`GAME_ROUNDS = 3`);
- порядок операций в рамках игрового цикла;
- механизм вывода вопроса и чтения ответа;
- условия успешного и неуспешного завершения.

На уровне архитектуры `pipeline.js` выполняет функцию управляющего ядра приложения. Его наличие позволяет проекту сохранять единообразие поведения независимо от того, сколько игровых режимов будет добавлено в дальнейшем.

#### Уровень регистрации и диспетчеризации: `src/game-registry.js`

Модуль `game-registry.js` инкапсулирует логику выбора игрового модуля по имени команды. Он содержит словарь доступных игр и экспортирует функцию `getGame(gameName)`, выполняющую проверку корректности входного имени.

Это решение устраняет хрупкость прямого обращения к объекту по строковому ключу из разных частей системы. В случае неизвестного имени игры формируется явная диагностическая ошибка `Unknown game: ...`, что делает контракт использования движка более строгим и понятным.

#### Уровень доменной логики: `src/games/`

Каждый файл в каталоге `src/games` отвечает за одну отдельную игру. Общий контракт игрового модуля следующий:

- функция правил игры (`*Rules`);
- функция генерации одного раунда (`*Round`).

Функция генерации раунда возвращает объект вида:

```js
{
  question: '... ',
  correctAnswer: '...'
}
```

Это означает, что конкретная игра отвечает только за построение содержательного состояния раунда, но не управляет самим сценарием общения с пользователем. Такое разделение повышает ясность модели и соответствует принципу локализации предметной логики.

#### Сервисный и инфраструктурный уровень

- `cli.js` изолирует получение пользовательского ввода.
- `greeting.js` централизует приветствие и получение имени пользователя.
- `game-over.js` централизует финальное сообщение.
- `utils.js` предоставляет общую функцию генерации случайных чисел.

### Внутренняя модель взаимодействия модулей

С точки зрения потока управления система функционирует следующим образом:

1. Пользователь вызывает одну из CLI-команд.
2. Соответствующий файл в `bin/` передаёт имя игры в `pipeline()`.
3. `pipeline()` получает описание игры через `getGame()`.
4. Выполняется приветствие пользователя.
5. На экран выводятся правила конкретной игры.
6. Движок трижды вызывает генератор раунда.
7. Для каждого раунда отображается вопрос и считывается ответ.
8. Ответ сравнивается с эталонным значением `correctAnswer`.
9. При ошибке игра немедленно завершается; при успехе происходит переход к следующему раунду.
10. После завершения выводится итоговое сообщение.

Данный поток характеризуется высокой линейностью, прозрачностью и отсутствием скрытых побочных ветвлений, что особенно ценно в учебных проектах, ориентированных на формирование системного мышления программиста.

### Характеристика игровых модулей

#### `brain-even-implementation.js`

Игра на определение чётности числа. Модуль содержит локальную функцию `isEven()` и формирует раунд на основе случайного числа.

#### `brain-calc-implementation.js`

Игра на вычисление арифметического выражения. Использует случайный выбор операции и двух операндов.

#### `brain-gcd-implementation.js`

Игра на поиск наибольшего общего делителя. Внутри реализован алгоритм Евклида.

#### `brain-progression-implementation.js`

Игра на восстановление скрытого элемента арифметической прогрессии. Модуль формирует последовательность, скрывает один элемент и ожидает правильное значение от пользователя.

#### `brain-prime-implementation.js`

Игра на определение простого числа. Модуль использует отдельную функцию `isPrime()` с проверкой делителей до квадратного корня.

### Технологические и инфраструктурные характеристики

Проект реализован на платформе Node.js и использует:

Такой стек полностью соответствует назначению проекта и не перегружен избыточными зависимостями.

### Характеристики проекта с точки зрения архитектуры 

К сильным сторонам реализованной в проекте архитектуры относятся:

- наличие централизованного игрового движка;

- хорошая расширяемость при добавлении новых игр.

## Техническая документация

### Назначение проекта

Проект `frontend-project-44` реализует набор консольных игр, направленных на тренировку базовых вычислительных и логических навыков. Приложение запускается через CLI-команды и взаимодействует с пользователем в интерактивном текстовом режиме.

### Состав игровых режимов

В проект входят следующие игры:

- `brain-even` — проверка числа на чётность;
- `brain-calc` — вычисление арифметического выражения;
- `brain-gcd` — поиск наибольшего общего делителя;
- `brain-progression` — восстановление скрытого элемента прогрессии;
- `brain-prime` — определение простого числа.

Дополнительно команда `brain-games` выводит приветствие и запрашивает имя пользователя.

### Контракт игрового модуля

Каждый игровой модуль должен экспортировать:

1. функцию правил игры;
2. функцию генерации одного раунда.

Объект раунда имеет следующий формат:

```js
{
  question: 'текст вопроса',
  correctAnswer: 'правильный ответ в строковом виде'
}
```

При добавлении новой игры необходимо соблюдать именно этот интерфейс, поскольку он используется движком `pipeline.js`.

### Правила подключения новой игры

Чтобы добавить новую игру в проект, необходимо:

1. Создать новый модуль в каталоге `src/games/`.
2. Экспортировать из него функцию правил и функцию генерации раунда.
3. Подключить игру в `src/game-registry.js`.
4. Создать соответствующий CLI-entrypoint в каталоге `bin/`.
5. Зарегистрировать новую команду в секции `bin` файла `package.json`.
6. При необходимости добавить удобную цель в `Makefile`.

### Команды проекта

Секции запуска определены в `package.json` и `Makefile`.

Основные CLI-команды:

- `brain-games`
- `brain-even`
- `brain-calc`
- `brain-gcd`
- `brain-progression`

### Общий сценарий игры

Во всех игровых режимах используется единая схема взаимодействия:

1. приложение приветствует пользователя и запрашивает имя;
2. отображаются правила выбранной игры;
3. последовательно выводятся вопросы;
4. пользователь вводит ответы в терминале;
5. для победы необходимо дать три правильных ответа подряд;
6. при первой ошибке игра завершается с сообщением о правильном ответе.

### Краткое назначение каждой игры

**brain-even**  
Нужно ответить `yes`, если число чётное, и `no`, если нечётное.

**brain-calc**  
Нужно вычислить результат арифметического выражения.

**brain-gcd**  
Нужно найти наибольший общий делитель двух чисел.

**brain-progression**  
Нужно определить скрытый элемент арифметической прогрессии.

**brain-prime**  
Нужно ответить `yes`, если число простое, и `no`, если составное.
